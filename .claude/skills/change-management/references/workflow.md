# 需求变更管理工作流详细参考

## 变更目录结构

```
output/{需求名}/changes/
├── CR-001_电商网点效能指标/
│   ├── 变更需求.md          ← Phase 2 产出
│   ├── 变更SQL.sql          ← Phase 3 产出
│   └── 变更审查报告.md      ← Phase 4 产出
├── CR-002_新增客户分层标签/
│   └── ...
└── ...
```

---

## 完整工作流

### Phase 1: 变更识别 (Change Identification)

**输入**：
- 原始需求文档（已交付版本）
- 新需求文档（变更请求）

**处理步骤**：
1. 读取原始需求文档，提取已有字段清单和逻辑
2. 读取新需求文档，提取新增/修改内容
3. 逐字段对比，识别：
   - 新增字段（原始需求中不存在）
   - 修改字段（逻辑/计算方式变化）
   - 删除字段（新需求中移除）
   - 逻辑变更（相同字段但计算规则不同）

**输出**：变更字段清单 + 逻辑差异说明

### Phase 3: 变更 SQL

**文件**：`CR-{NNN}_{简称}/变更SQL.sql`

**结构要求**：

1. **文件头注释**：
```sql
-- ============================================================
-- CR-{NNN}_{变更简称}
-- 变更类型: {类型}
-- 日期: {YYYY-MM-DD}
-- 提出人: {业务方}
-- 执行人: {开发人员}
-- ============================================================
```

2. **修改的TMP表**（完整重建）：
```sql
-- 1. 修改TMP表: {表名}
DROP TABLE IF EXISTS {库名}.{表名};
CREATE TABLE {库名}.{表名} AS
SELECT ...
```

3. **新增的TMP表**：
```sql
-- 2. 新增TMP表: {表名}
DROP TABLE IF EXISTS {库名}.{表名};
CREATE TABLE {库名}.{表名} AS
SELECT ...
```

4. **修改的ADS表**（追加字段片段）：
```sql
-- 3. 修改ADS表: {表名}
-- 追加到 SELECT:
--   t_new.{字段1},  -- [CR-NNN] {字段说明}
--   t_new.{字段2}   -- [CR-NNN] {字段说明}
-- 追加 JOIN:
--   LEFT JOIN {新TMP表} t_new
--     ON t_main.{关联键} = t_new.{关联键}
```

5. **重跑顺序**：
```sql
-- 4. 重跑顺序:
--   1) {TMP表1}
--   2) {TMP表2}
--   3) {ADS表}
```

**命名规范**：
- 新增TMP表：`tmp_{库名}.tmp_{需求编号}_{主题}`
- 变更字段注释：`-- [CR-NNN] {字段说明}`

---

### Phase 4: 变更审查

**文件**：`CR-{NNN}_{简称}/变更审查报告.md`

**审查维度**：

| 维度 | 检查项 | 检查方法 |
|------|--------|----------|
| **逻辑正确性** | 新增字段逻辑是否符合需求定义 | 对比变更需求文档 |
| **回归风险** | 已有字段是否受影响 | 对比变更前后SELECT列表 |
| **性能影响** | 新增扫描量、JOIN数量 | 分析SQL执行计划 |
| **数据一致性** | 新增字段与已有字段交叉验证 | 编写验证SQL |
| **待确认问题** | 所有问题是否已确认 | 检查变更需求文档 |

**审查结论**：
- ✅ **通过** — 所有检查项通过，可执行
- ⚠️ **条件通过** — 存在待确认问题，确认后可执行
- ❌ **不通过** — 存在逻辑错误或高风险，需修改后重新审查

---

### Phase 5: 合并执行

**步骤**：

1. **合并到主SQL**：
   - 将变更SQL中的SELECT片段追加到主SQL
   - 将变更SQL中的JOIN片段追加到主SQL
   - 确保字段顺序一致

2. **更新DDL**：
   - 在表结构文件中追加新字段
   - 更新字段注释

3. **更新DQC**：
   - 新增字段的唯一性测试
   - 新增字段的非空率测试
   - 新增字段的枚举值测试

4. **更新文档**：
   - 同步更新 Design.md
   - 同步更新知识沉淀.md

5. **重跑验证**：
   - 按重跑顺序执行SQL
   - 运行回归验证清单

---

### Phase 6: 归档

1. **更新交付总报告**：
   - 在"开发过程问题修复记录"中追加变更记录
   - 记录CR编号、变更内容、影响范围

2. **更新提效看板**：
   - 统计变更工作量（SQL行数、耗时）

3. **更新语义模型**：
   - 如有新业务域/实体变更，更新 domain JSON

## 回归验证清单

每次变更后**必须**执行以下验证：

| # | 验证项 | SQL模板 | 预期结果 |
|---|--------|---------|----------|
| 1 | 主键唯一性不变 | `SELECT COUNT(*) = COUNT(DISTINCT {主键}) FROM {ADS表}` | true |
| 2 | 已有字段值不变 | 对比变更前后原有字段值（抽样） | 一致 |
| 3 | 新增字段非空率 | `SELECT COUNT({新字段})/COUNT(*) FROM {ADS表}` | > 阈值 |
| 4 | 新增字段枚举值 | `SELECT DISTINCT {标签字段} FROM {ADS表}` | 符合定义 |
| 5 | 数据行数不变 | `SELECT COUNT(*) FROM {ADS表}` | 与变更前一致 |

**回归验证SQL示例**：

```sql
-- 回归验证: CR-{NNN}
-- 日期: {YYYY-MM-DD}

-- 1. 主键唯一性
SELECT
    '主键唯一性' as check_item,
    CASE WHEN COUNT(*) = COUNT(DISTINCT emp_code) THEN 'PASS' ELSE 'FAIL' END as result
FROM ads.ads_sample_business_report
WHERE inc_day = '${bizdate}';

-- 2. 数据行数对比
SELECT
    '数据行数' as check_item,
    COUNT(*) as row_count,
    {变更前行数} as expected_count,
    CASE WHEN COUNT(*) = {变更前行数} THEN 'PASS' ELSE 'FAIL' END as result
FROM ads.ads_sample_business_report
WHERE inc_day = '${bizdate}';

-- 3. 新增字段非空率
SELECT
    '新增字段非空率' as check_item,
    COUNT({新字段})/COUNT(*) as non_null_rate,
    CASE WHEN COUNT({新字段})/COUNT(*) > 0.9 THEN 'PASS' ELSE 'FAIL' END as result
FROM ads.ads_sample_business_report
WHERE inc_day = '${bizdate}';

-- 4. 新增字段枚举值
SELECT DISTINCT {标签字段}
FROM ads.ads_sample_business_report
WHERE inc_day = '${bizdate}'
ORDER BY {标签字段};
```

## 变更类型详解

### 新增字段

**特征**：添加新的计算字段
**影响**：新增TMP表 + 修改ADS表
**示例**：新增"电商网点效能"指标

**处理要点**：
- 创建新的TMP表存储中间计算结果
- 在ADS表中追加字段
- 追加JOIN关联

### 逻辑修改

**特征**：修改已有字段的计算逻辑
**影响**：修改TMP表 + 修改ADS表
**示例**：修改"效能等级"的标签区间

**处理要点**：
- 完整重建受影响的TMP表
- 确保修改后的逻辑与需求一致
- 回归验证原有字段值

### 删除字段

**特征**：移除不再需要的字段
**影响**：修改ADS表SELECT
**示例**：废弃"临时指标"

**处理要点**：
- 从SELECT列表中移除字段
- 从DDL中移除字段定义
- 从DQC中移除相关测试

### 数据源变更

**特征**：更换或新增数据源
**影响**：可能影响多个TMP表
**示例**：切换维表来源

**处理要点**：
- 评估影响范围
- 更新所有受影响的TMP表
- 验证数据一致性

### 架构调整

**特征**：TMP表拆分/合并
**影响**：重构TMP层
**示例**：新增DWS层

**处理要点**：
- 完整重构TMP层
- 更新所有下游依赖
- 全面回归验证

## 常见问题处理

### Q1: 变更审查不通过怎么办？

**处理流程**：
1. 记录不通过原因到审查报告
2. 修改变更SQL
3. 重新执行审查
4. 通过后才能合并

### Q2: 多个变更同时进行如何处理？

**处理原则**：
- 每个变更独立建档（独立CR目录）
- 按CR编号顺序合并
- 每次合并后执行回归验证
- 如有冲突，后合并的变更需重新审查

### Q3: 变更合并后发现回归问题怎么办？

**处理流程**：
1. 立即回滚到变更前版本
2. 分析回归问题原因
3. 修改变更SQL
4. 重新审查和合并

### Q4: 如何追踪变更历史？

**追踪方式**：
- 所有变更记录在 `output/{需求名}/changes/` 目录
- 交付总报告中记录所有CR编号和变更摘要
- 变更SQL中用 `-- [CR-NNN]` 注释标记字段来源
